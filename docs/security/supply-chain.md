# Supply Chain Security & Release Process

This document outlines the controls and roadmaps for securing the Agent Knowledge Compiler and Control Plane (AKCP) supply chain. It acts as a guide for contributors and reviewers to ensure that our open-source software delivery is resilient against supply-chain attacks.

## Current CI/CD Controls

1. **Pinned Actions**: We pin critical GitHub Actions to specific SHA hashes (e.g., ctions/checkout@<sha>) rather than mutable tags (@v4) to prevent upstream hijacking of workflows.
2. **Immutable Dependencies**: Our lockfile (pnpm-lock.yaml) must always be frozen during CI execution (pnpm install --frozen-lockfile) to prevent unintended dependency resolution variations.
3. **Automated Scanning**:
   - **Dependabot**: Enabled for automated vulnerable dependency updates (both npm and GitHub Actions ecosystems).
   - **CodeQL**: Runs automatically on Pull Requests and via schedule to detect static analysis vulnerabilities.
   - **Dependency Review**: Blocks PRs from introducing net-new vulnerabilities.
4. **Least Privilege Workflows**: All .github/workflows/*.yml must explicitly define a permissions: block at the top-level (e.g., contents: read), and only elevate rights for specific jobs that strictly require them (e.g., packages: write in elease.yml).

## Release Readiness Process

To release a new version of AKCP, maintainers MUST follow this checklist:

1. **Test Suite Passage**: Ensure all unit, contract, and security test suites are passing on main.
2. **Bundle Verification**: The core pnpm akcp validate, pnpm akcp compile, and pnpm akcp conformance run targets must successfully execute against standard example data.
3. **No Committed Secrets**: Use GitHub Advanced Security secret scanning to verify no API keys, internal tokens, or actual PII made it into the release commit.
4. **Tag Generation**: Create a signed Git tag following Semantic Versioning (e.g., 1.0.0).
5. **NPM Publication**:
   - The release pipeline will build the packages.
   - For publishing to NPM, 2FA must be enforced for all organization members.
   - In the future, we will utilize --provenance via NPM to attach build attestations natively.

## Roadmap: SBOM, Provenance, and Signing

While our current controls are robust, we intend to achieve full SLSA compliance in the future:

1. **Software Bill of Materials (SBOM)**:
   - *Goal*: Generate an automated SBOM (SPDX or CycloneDX format) as a build artifact on every release tag.
   - *Implementation*: Integrate @cyclonedx/cyclonedx-npm or GitHub's Dependency Submission API into the release workflow.
2. **Artifact Provenance (SLSA Level 3)**:
   - *Goal*: Prove that artifacts published on NPM were generated by our exact CI runner, from a specific commit.
   - *Implementation*: Turn on provenance: true in NPM publish commands, leveraging GitHub Actions OIDC integration with npmjs.com.
3. **Commit & Tag Signing**:
   - *Goal*: Mandate that all commits and release tags must be signed by trusted maintainers (GPG or SSH signing).

> [!IMPORTANT]
> The principles detailed here apply across all domains inside AKCP (e.g., Career, IT Operations, Customer Support). No new package or binary should bypass these validation phases.
